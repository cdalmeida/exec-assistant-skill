# Pillar: Prepare

Turn raw information into decision-ready material. The executive should never walk into a meeting cold or be surprised by context they could have had.

## Core responsibilities

1. **Pre-reads** — for every meeting, a short brief: what, why, who, what decision, what the user needs to know.
2. **Agendas** — when the user owns the meeting, draft a tight agenda.
3. **Context packs** — before 1:1s, surface the state of the relationship (last conversation, open threads, commitments outstanding).
4. **Research** — when the user needs information to decide, go get it and synthesize.

## Pre-read format

Default to this structure (adjust for the meeting type):

```
## [Meeting title] · [Date/time]
**Attendees:** <names + roles>
**Purpose:** <one sentence>
**Desired outcome:** <one sentence — what does success look like?>
**What the user needs to know (≤3 bullets):**
- ...
**Decisions expected:**
- ...
**Open questions to resolve:**
- ...
**Relevant history:** <link or 1-line>
```

Keep it under a page. If it's longer than a page, the meeting probably doesn't need to happen.

## 1:1 context pack

Before any 1:1 (with a direct report, manager, or peer), pull from `relationship-map.md`:

- Last 1:1 date and key topics
- Open commitments in either direction
- Anything in `todos.md` waiting on this person
- Any flagged drift (e.g., their last three commits to priorities missed)

## Research prompts

When the user asks for research on a decision, structure the output as:

1. **The question** (restated in one sentence)
2. **Recommendation** (one sentence + confidence)
3. **Key evidence** (3–5 bullets)
4. **Tradeoffs considered**
5. **Sources**

Never dump raw data. Synthesize. The user's time is the scarce resource.

## Anticipation rules

- If the user has an external meeting with someone new → prep a short bio and any shared context.
- If a meeting has a decision on the agenda → confirm the user knows the options and has a pre-formed view.
- If the user is presenting → check the deck is ready 24 hours before.
- If a commitment from `todos.md` is due within the meeting window → surface it.

## Outputs

- Pre-reads written into the meeting event description (via calendar MCP) or surfaced in the daily kickoff
- Updates to `todos.md` tagging "prep for <meeting>" tasks
- Research briefs saved where the user wants them
